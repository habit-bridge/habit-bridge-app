# Habit Bridge Demo Server — 요구사항 문서 (MVP)

## 1. 개요

XRP Ledger(XRPL)의 **Escrow**를 활용해 사용자가 습관형성 챌린지에 참여할 때 **보증금(XRP)을 에스크로로 예치**하고, 챌린지 결과(성공/실패)에 따라 **기부로 귀속**되는 서버를 NestJS 기반으로 구현한다.

- **성공**: 사용자 이름으로 기부
- **실패**: 서비스 이름으로 기부

본 문서는 MVP 범위에서 필요한 기능 요구사항과 핵심 플로우, 데이터/정책, 외부 연동 지점을 정의한다.

## 2. 용어

- **사용자(User)**: 서비스 계정 주체. 외부 XRP 지갑을 이미 보유.
- **챌린지(Challenge)**: 사용자가 생성하는 습관형성 단위.
- **참여(Participation)**: 사용자의 챌린지 참가 레코드. 보증금 에스크로와 1:1로 연결.
- **인증(Verification)**: 참여 중 주기적으로 제출하는 “날짜가 포함된 사진” 증빙.
- **보증금(Deposit)**: 참여 시 예치하는 XRP 금액.
- **기부(Donation)**: 챌린지 결과에 따른 기부 이벤트(집계/랭킹의 근거 데이터).
- **Xaman(Xumm)**: XRPL 서명 UX를 제공하는 외부 월렛/SDK(또는 직접 XRPL 서명 흐름).

## 3. 핵심 유저 시나리오

### 3.1 회원가입/로그인

1) 사용자가 회원가입  
2) 로그인하여 액세스 토큰을 발급받음  
3) (선택) 기부 표기용 이름(display name)을 설정/수정

### 3.2 챌린지 생성

1) 로그인한 사용자가 챌린지를 생성  
2) 시작일/기간/인증 주기 등의 정책에 따라 챌린지 종료일이 결정됨  
3) 챌린지는 공개 조회 가능(정책에 따라 변경 가능)

### 3.3 챌린지 참여 + XRP 에스크로 예치

1) 사용자가 **종료되지 않은 챌린지** 목록을 조회  
2) 참여 버튼 클릭  
3) 서버가 **EscrowCreate 서명을 위한 트랜잭션 템플릿**(또는 Xaman payload)을 생성  
4) 사용자는 **Xaman 또는 XRPL 지갑으로 서명**  
5) 서버는 서명 결과를 확인하고 참여 상태를 “진행 중”으로 전환

### 3.4 사진 인증 업로드

1) 참여 중인 사용자는 주기마다 **날짜가 포함된 사진**을 업로드  
2) 서버는 업로드를 참여/주기 슬롯에 귀속  
3) 종료일까지 모든 인증 조건을 만족하면 성공

### 3.5 종료/정산/기부 귀속

1) 챌린지 종료 시점에 참여자별 성공/실패 판정  
2) 성공 시 “사용자 이름으로 기부”, 실패 시 “서비스 이름으로 기부”로 Donation 이벤트 기록  
3) (선택) 실제 온체인 기부 송금이 있다면 비동기 처리/재시도

### 3.6 랭킹

- **기부 많이 한 사람 TOP 10**
- **챌린지 성공 많이 한 사람 TOP 10**

## 4. 기능 요구사항

### 4.1 인증/계정

- **회원가입**
  - 이메일/아이디/비밀번호 정책은 MVP에서 단순화 가능(예: 이메일+비밀번호)
  - 비밀번호는 해시로 저장
- **로그인**
  - 성공 시 액세스 토큰 발급(JWT 등)
- **프로필**
  - 기부 표기용 이름(예: `displayName`) 저장/수정
  - (선택) 사용자 XRPL 주소(r-address) 연결

### 4.2 챌린지(생성/조회)

챌린지 생성 시 아래 항목을 입력받는다.

- **시작일** (date)
- **기간** (몇 주, integer)
- **제목** (string)
- **소개** (string)
- **인증 주기** (예: 주 1회/주 3회/매일 등 정책화)
- **인증 방법 설명** (string)
- **보증금** (XRP 금액, decimal)

추가 정책:

- 사용자는 **종료되지 않은 챌린지**에만 참여할 수 있다.
- 챌린지는 시작 전/진행 중/종료 상태를 가진다.

### 4.3 참여(Participation)

- 사용자는 특정 챌린지에 참여 가능(챌린지가 종료되지 않은 경우)
- 동일 사용자의 동일 챌린지 **중복 참여는 불가**
- 참여는 다음 상태를 최소로 가진다(예시):
  - `PENDING_DEPOSIT` (에스크로 예치 전/대기)
  - `DEPOSIT_SUBMITTED` (서명 제출됨)
  - `ACTIVE` (에스크로 확인 완료, 진행 중)
  - `COMPLETED_SUCCESS` (성공)
  - `COMPLETED_FAIL` (실패)
  - `CANCELLED` (중도 취소/무효 처리)

### 4.4 XRPL 에스크로(필수)

요구사항:

- 사용자는 외부 XRP 지갑을 이미 보유
- **Xaman 또는 XRPL 연동으로 EscrowCreate 서명 흐름 필요**
- 참여 시 보증금을 **에스크로 결제(예치)** 해야 함

서명/연동 흐름(권장 API 분리):

- **Prepare(준비)**: 서버가 EscrowCreate 트랜잭션 템플릿(또는 Xaman payload)을 생성하고, 참여 레코드에 요청 식별자/해시를 저장한다.
- **Submit(제출/확인)**: 사용자가 서명 결과(tx blob 또는 payload 결과)를 서버에 전달하면 서버가 필수 필드(금액/수신자/만료 조건 등)를 검증하고 제출/확인한다.

검증 최소 기준(서버가 반드시 확인해야 하는 항목):

- 대상 챌린지/참여와 매칭되는 요청인지
- 보증금 금액(XRP drops 변환 포함)이 챌린지 조건과 일치하는지
- Escrow 수신자/조건(서비스 정책)이 서버가 생성한 값과 일치하는지
- 이미 처리된 제출인지(멱등성)

※ MVP에서는 **Devnet/Testnet에서 플로우가 재현되는 것**을 완료 기준으로 한다(메인넷은 별도).

### 4.5 인증(사진 업로드)

요구사항:

- 참여 중인 챌린지에 **날짜가 들어간 사진을 업로드해 인증**

기능:

- 사진 업로드(파일 타입/용량 제한)
- 업로드 파일을 참여(Participation) 및 해당 인증 슬롯(주기/날짜)에 귀속
- 각 참여에 대해 “현재까지 완료한 인증 수/남은 인증 수” 조회 가능

검증 방식(정책 옵션, MVP 우선순위는 낮음):

- (MVP) 서버 업로드 시각을 기준으로 슬롯 귀속 + 최소한의 수동/간단 검증
- (후속) OCR/EXIF/모더레이션 큐 등으로 “날짜 포함”의 신뢰성 강화

### 4.6 성공/실패 판정

요구사항:

- 종료일까지 모든 인증 조건을 만족하면 성공
- 성공 시 사용자 이름으로 기부
- 실패 시 서비스 이름으로 기부

기능:

- 챌린지 종료 시점에 참여별 성공/실패를 판정하는 정산 로직
- 결과를 변경 불가하게 기록(멱등, 재실행 안전)

### 4.7 기부 및 랭킹

요구사항:

- 기부 많이 한 사람 TOP 10
- 챌린지 성공 많이 한 사람 TOP 10

기능:

- Donation 이벤트 로그 저장(누구 이름으로/얼마나/어떤 참여·챌린지에서)
- 집계 API 제공:
  - **Donation TOP 10**: 사용자 기준 누적 기부액
  - **Success TOP 10**: 사용자 기준 성공 횟수

## 5. 비기능 요구사항(최소)

- **보안**
  - 비밀번호 해시 저장
  - 업로드 파일의 타입/크기 제한
  - 인증 필요 API는 토큰 기반 보호
- **신뢰성**
  - XRPL 제출/확인은 네트워크 오류를 고려해 재시도 가능(비동기 Job 권장)
  - 정산/기부 기록은 멱등 처리
- **감사 가능성**
  - 참여 상태 변경, 서명 제출, 정산 결과 등을 이벤트/로그로 남김

## 6. MVP 범위(명시)

MVP에서 반드시 되는 것:

- 회원가입/로그인
- 챌린지 생성/조회
- 종료되지 않은 챌린지 참여
- 참여 시 **EscrowCreate 서명 흐름**(Xaman 또는 XRPL)
- 사진 업로드 인증
- 성공/실패 판정 및 Donation 이벤트 기록
- TOP 10 랭킹 2종 제공

MVP에서 후순위(선택):

- 날짜 포함 사진의 자동 진위 판별(OCR/EXIF/부정 방지)
- 온체인 기부 송금의 완전 자동화(우선은 오프체인 Donation 로그로 시작 가능)
- 관리자(분쟁 처리/수동 승인) UI 및 권한 시스템
